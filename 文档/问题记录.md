# Phase 3 前端启动报错：I/O operation on closed file

## 报错现象

启动 Streamlit 聊天界面后，在浏览器中输入问题并提交，页面报错：

```
ValueError: I/O operation on closed file.
  File "app.py", line 70, in <module>
    candidates = retriever.search(query, top_k=10)
  File "app\rag\retriever.py", line 61, in search
    results, _scores = self._bm25.retrieve(query_tokens, k=bm25_k)
  File "bm25s\__init__.py", line 902, in retrieve
    out = tqdm(map(topk_fn, query_tokens), **tqdm_kwargs)
  File "tqdm\std.py", line 1096, in __init__
    self.sp = self.status_printer(self.fp)
  File "tqdm\std.py", line 449, in status_printer
    getattr(sys.stdout, 'flush', lambda: None)()
ValueError: I/O operation on closed file
```

## 根因分析

两个因素叠加导致：

### 1. `sys.stdout` 被重复包装

`app.py` 顶部写了一段 Windows 中文兼容代码：

```python
sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8", errors="replace")
```

Streamlit 在启动时已经用自己的机制管理 `sys.stdout`（内部使用 `contextlib.redirect_stdout` 等）。再次包装 `sys.stdout` 会打破 Streamlit 的输出管道，导致后续写入时文件句柄已关闭。

### 2. `tqdm` 进度条与 Streamlit 冲突

`bm25s` 库内部使用 `tqdm` 打印进度条（`BM25S Retrieve: 0%`）。`tqdm` 需要实时写入并 flush `sys.stdout`。当 `sys.stdout` 被错误包装后，`tqdm` 调用 `sys.stdout.flush()` 时发现文件已关闭，抛出 `ValueError`。

**调用链：** 用户提交问题 → `retriever.search()` → `bm25.retrieve()` → 内部 `tqdm` → `sys.stdout.flush()` → 文件已关闭 → 报错

## 解决方案

### 修复 1：移除 `app.py` 中的 `sys.stdout` 包装

```python
# 删除这行（Streamlit 自己管理 UTF-8 输出，无需手动设置）
sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8", errors="replace")
```

### 修复 2：`retriever.py` 中禁用 bm25s 进度条

```python
# _build_bm25() 中
self._bm25.index(tokens, show_progress=False)  # 加 show_progress=False

# search() 中
results, _scores = self._bm25.retrieve(query_tokens, k=bm25_k, show_progress=False)
```

## 涉及文件

| 文件 | 修改 |
|------|------|
| `app.py` | 删除 `sys.stdout` 重包装（2 行） |
| `app/rag/retriever.py` | `index()` 和 `retrieve()` 加 `show_progress=False` |

## 教训

- **不要在 Streamlit 应用中手动改 `sys.stdout`**。Streamlit 有自己完整的 UTF-8 处理链路，手动干预反而破坏其机制。
- **第三方库的进度条（tqdm）在 Streamlit 中通常不兼容**。应当通过参数禁用以避免 stdout 冲突。
- **`streamlit run` 和普通 `python` 运行环境不同**。同样的代码在 CLI 脚本（如 `verify_phase3.py`）中可能正常，但在 Streamlit 上下文中需要适配。
