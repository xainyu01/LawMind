# 法律 RAG 系统 — 详细项目介绍

## 一、项目概述

**法律 RAG 系统**是一个面向中国法律领域的**检索增强生成（Retrieval-Augmented Generation）**智能问答系统。该系统能够理解用户的法律问题，从知识库中精准检索相关法律条文，并结合大语言模型生成专业、有据可查的法律回答。

### 核心价值

- **法条溯源**：每次回答都标注完整的法律出处（法律名称 + 条/款/项），确保回答可验证
- **反幻觉机制**：严格限制只能引用检索到的法条，杜绝 LLM 编造法律条文
- **多意图覆盖**：支持法条查询、案例分析、法律知识问答、合同审查等多种法律场景

---

## 二、核心功能特性

### 1. 智能问答能力

| 功能 | 说明 |
|------|------|
| **法条查询** | 精确查询法律条文内容，如"民法典第1043条是什么" |
| **案例分析** | 分析法律案例、判例，如"借款利息过高怎么判" |
| **法律知识问答** | 解答法律概念、原则，如"什么是诉讼时效" |
| **合同审查** | 辅助分析合同条款，如"租赁合同有没有不公平条款" |
| **多轮对话** | 支持上下文记忆的连续对话 |

### 2. 混合检索引擎

```
用户查询 → BM25 关键词检索（精确匹配）
         → 向量相似度检索（语义理解）
         → RRF 融合排序
         → Cross-Encoder 重排序
         → MMR 去重
         → 返回 Top-K 结果
```

- **BM25 检索**：适合精确的法条编号查询（如"第1043条"）
- **向量检索**：适合语义模糊的法律咨询（如"夫妻之间有什么义务"）
- **RRF 融合**：Reciprocal Rank Fusion 算法融合两种检索结果
- **Cross-Encoder 重排序**：使用 `bge-reranker-base` 模型精排，法律场景下"找对法条"比"找到相似内容"更关键
- **MMR 去重**：Maximal Marginal Relevance 算法保证结果多样性，避免重复法条

### 3. 对话记忆系统

- **短期记忆**：Redis List 存储最近 10 轮对话历史
- **长期记忆**：Redis Hash 存储提取的关键信息（法律领域、已引用法条、用户陈述事实）
- **上下文注入**：将历史对话和关键信息注入 Prompt，保持法律推理连贯性

### 4. 语义缓存

- **相似度阈值**：0.92 以上的相似问题直接返回缓存答案
- **差异化 TTL**：
  - 法条查询：7 天（法律相对稳定）
  - 案例分析：30 天（案例不过时）
  - 合同审查：1 天（合同场景变化快）
- **法条时效性**：自动过滤已废止法条的缓存

### 5. 意图识别路由

使用 LLM Prompt 进行意图分类（降级方案为关键词匹配）：

| 意图类型 | 路由策略 | 示例 |
|----------|----------|------|
| `statute_lookup` | 向量检索 + 法条原文返回 | "民法典第1043条是什么" |
| `case_analysis` | RAG + 相似案例检索 | "借款利息过高怎么判" |
| `legal_qa` | RAG + 知识库 | "什么是诉讼时效" |
| `contract_review` | 指定模板 + 条款比对 | "租赁合同审查" |
| `chitchat` | 固定话术返回 | "你好"、"你是谁" |

### 核心处理流程

```
1. 用户输入问题
   ↓
2. 意图识别（LLM Prompt / 关键词降级）
   ↓
3. 语义缓存检查（embedding 相似度 > 0.92）
   ├── 命中 → 直接返回缓存答案
   └── 未命中 ↓
4. 混合检索（BM25 + 向量 + RRF 融合）
   ↓
5. Cross-Encoder 重排序
   ↓
6. 相关性过滤（score > 0.1）
   ↓
7. MMR 去重（保证多样性）
   ↓
8. 注入对话记忆上下文
   ↓
9. LLM 生成答案（DeepSeek API）
   ↓
10. 保存对话记忆 + 更新缓存
   ↓
11. 返回答案 + 参考法条
```

## 六、核心模块详解

### 1. 文档处理模块 (`document_loader.py`)

**功能**：解析法律文档并进行智能分段

**支持的分段模式**：
- **默认模式**：LangChain `RecursiveCharacterTextSplitter`，chunk_size=500，chunk_overlap=50
- **法律结构化分段**：
  - 法律法规：按"第X条"分段，保持法条完整性
  - 判决书：按"本院认为"、"判决如下"等关键段落分段
  - 合同：按条款分段

**分隔符优先级**：`\n\n` → `\n` → `。` → `；` → `，` → ` `

### 2. 嵌入模型模块 (`embedding.py`)

**功能**：将文本转换为向量表示

**技术细节**：
- 模型：`BAAI/bge-small-zh`（~100MB）
- 设备：CUDA GPU 加速
- 归一化：启用 `normalize_embeddings`
- 缓存：模型下载到 `./models` 目录

### 3. 向量存储模块 (`vector_store.py`)

**功能**：管理 ChromaDB 向量数据库

**核心操作**：
- `add_documents()`：批量入库文档（文本 + 向量 + 元数据）
- `search()`：向量相似度检索，支持元数据过滤
- `get_all_documents()`：获取所有文档（用于 BM25 索引构建）
- `get_embeddings_by_ids()`：根据 ID 获取向量（用于 MMR 计算）

### 4. 混合检索模块 (`retriever.py`)

**功能**：BM25 + 向量混合检索，RRF 融合，MMR 去重

**检索流程**：
1. **BM25 检索**：使用 jieba 分词，bm25s 库构建索引，支持持久化缓存
2. **向量检索**：调用 ChromaDB 相似度搜索，支持过滤已废止法条
3. **RRF 融合**：`score = 1/(k+rank)`，k=60
4. **MMR 去重**：`MMR = λ * relevance - (1-λ) * max_similarity`，λ=0.5

**性能优化**：
- BM25 索引首次构建后持久化到磁盘
- 后续启动直接加载，减少启动时间

### 5. 重排序模块 (`reranker.py`)

**功能**：使用 Cross-Encoder 对检索结果精排

**技术细节**：
- 模型：`BAAI/bge-reranker-base`（~400MB）
- 输入：(query, document) 配对
- 输出：相关性分数，按分数降序排列

### 6. 生成器模块 (`generator.py`)

**功能**：调用 DeepSeek API 生成法律回答

**核心特性**：
- **法律系统 Prompt**：严格要求引用法条出处，禁止编造
- **意图分类**：LLM Prompt 分类（降级为关键词匹配）
- **流式生成**：支持逐 token 输出，提升用户体验
- **闲聊处理**：预设固定话术，无需调用 LLM

### 7. 对话记忆模块 (`memory.py`)

**功能**：管理多轮对话上下文

**存储结构**：
- **短期记忆**（Redis List）：存储最近 10 轮对话（user + assistant）
- **长期记忆**（Redis Hash）：
  - `law_domains`：涉及的法律领域
  - `cited_statutes`：已引用的法条
  - `user_facts`：用户陈述的关键事实
  - `last_intent`：最近意图

### 8. 语义缓存模块 (`cache.py`)

**功能**：基于 embedding 相似度的 LLM 回答缓存

**工作流程**：
1. 计算查询的 embedding
2. 在缓存索引中查找相似度 > 0.92 的条目
3. 命中则直接返回缓存答案
4. 未命中则走 RAG 流程，完成后存入缓存

**差异化 TTL**：

| 意图 | TTL | 原因 |
|------|-----|------|
| 法条查询 | 7 天 | 法律相对稳定 |
| 案例分析 | 30 天 | 案例不过时 |
| 法律问答 | 7 天 | 知识相对稳定 |
| 合同审查 | 1 天 | 场景变化快 |

## 七、API 接口

### 1. 问答接口

**POST** `/api/v1/chat`

**请求体**：
```json
{
  "query": "夫妻之间有什么义务？",
  "session_id": "user_123",
  "history": null
}
```

**响应体**：
```json
{
  "answer": "根据《中华人民共和国民法典》第1043条，夫妻应当互相忠实，互相尊重，互相关爱...",
  "sources": [
    {
      "content": "第一千零四十三条 夫妻应当互相忠实，互相尊重，互相关爱...",
      "source": "民法典节选.txt",
      "score": 0.8523
    }
  ],
  "intent": "statute_lookup",
  "cached": false
}
```

### 2. 健康检查

**GET** `/health`

```json
{
  "status": "ok"
}
```

### 3. 监控指标

**GET** `/metrics`

返回 Prometheus 格式的指标数据：
- `rag_requests_total`：请求总数（按意图和状态分类）
- `rag_request_duration_seconds`：请求耗时
- `rag_retrieval_duration_seconds`：检索耗时
- `rag_llm_duration_seconds`：LLM 生成耗时
- `rag_cache_hits_total`：缓存命中次数
- `rag_intent_distribution`：意图分布

- **内网限制**：`/metrics` 端点仅允许内网访问

## 十三、技术亮点

- **Docker 容器化**：一键部署，环境一致性
- **Nginx 反向代理**：负载均衡、SSL 终止、限流
- **Prometheus 监控**：全链路指标采集
- **结构化日志**：便于问题排查与分析

### 4. 用户体验
