## 二、技术选型建议（兼顾实习面试含金量）

| 层次              | 技术选项                                        | 理由                                       |
| ----------------- | ----------------------------------------------- | ------------------------------------------ |
| **前端**          | Python版用 Streamlit；Java版用 Vue + ElementUI  | 快速出界面，Vue你有基础                    |
| **API服务**       | Python：FastAPI；Java：SpringBoot + LangChain4j | FastAPI异步性能好，LangChain4j展示Java能力 |
| **LLM**           | DeepSeek / 通义千问 / OpenAI API                | 国内稳定选DeepSeek，便宜够用               |
| **Embedding模型** | text-embedding-3-small / bge-large-zh           | 中文检索推荐bge                            |
| **向量数据库**    | Milvus (Docker部署) 或 Chroma                   | Chroma轻量上手快，Milvus更工程化           |
| **文档解析**      | Unstructured / PyPDF2 / python-docx             | 支持PDF、Word、TXT                         |
| **缓存**          | Redis Stack (含RediSearch)                      | 做语义缓存和会话记忆，你正在学             |
| **消息队列**      | Redis Stream 或 RabbitMQ                        | 异步处理文档、转人工通知                   |
| **监控**          | Prometheus + Grafana (可选)                     | 面试时提一句“预留监控接口”很专业           |

## 三、核心模块详细设计

### 1. 文档处理管道（最基础的模块）
**流程：** 上传 → 解析 → 分段 → Embedding → 存入向量库  
- **分段策略**：按语义分段（LangChain的`RecursiveCharacterTextSplitter`），重叠窗口防止截断。
- **元数据保留**：每个chunk记录来源文档名、页码，方便回答时展示出处。
- **异步处理**：用户上传后，FastAPI返回“处理中”，用Celery/Redis Stream异步解析，完成后通知用户。

### 2. RAG检索与生成核心
**标准流程：**  
用户问题 → 向量化 → 向量库Top-K检索 → 相似度过滤 → 拼接上下文 → 注入Prompt → LLM生成答案  
- **检索优化**：不能用简单余弦相似度就算完。要加入**MMR（最大边际相关）**，保证召回结果多样且不冗余。
- **混合检索**：关键词BM25 + 向量相似度，各召回5条，再融合排序。这点能讲出来就比90%的人强。
- **重排序**：用Cross-Encoder模型（如bge-reranker）对初召回的文档重新打分，大幅提升准确率。

### 3. 对话记忆管理
你已有的Redis现在派上用场：
- 用**Redis List**保存每个`session_id`的最近10轮对话。
- 关键信息（如用户姓名、工号、问题焦点）提取后存入**Redis Hash**，作为长期记忆。
- 记忆注入Prompt：把最近对话和长期信息写入System Prompt，让Agent记住上下文。

### 4. 智能路由与意图识别
这是Agent自动决策的部分：
- 用户说“转人工”，直接触发转接流程（WebSocket通知坐席，你JavaWeb经验可以用上）。
- 问普通知识，走RAG。
- 问“你是谁”“你能干嘛”，用固定话术回复。
- 用一个小模型或一个Prompt先做意图分类，再分发。

### 5. 语义缓存
你正在学的RediSearch派上用场：
- 用户问题向量化 → 在Redis中搜索相似度>0.92的缓存 → 命中直接返回答案，未命中走RAG后存入缓存。
- 可以设置不同领域的缓存时间：政策类缓存1天，闲聊不缓存。

### 6. 可观测与评估
- 记录每次问答的**用户反馈**（点赞/点踩）。
- 离线用RAGAS框架评估：答案忠实度、上下文相关性。
- 面试时说“我们做了评测闭环，持续优化”，直接展示工程完整性。
