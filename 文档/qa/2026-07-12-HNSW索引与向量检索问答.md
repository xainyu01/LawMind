# HNSW 索引与向量检索问答

---

## 一、HNSW 是什么？

### Q: HNSW 是什么？在系统哪一层？

**HNSW**（Hierarchical Navigable Small World，分层可导航小世界图）是一种**近似最近邻（ANN）搜索算法**，用于在高维向量空间中快速找到与查询向量最相似的 Top-K 个向量。

**在系统中的位置**：

```
┌─────────────────────────────────────────────────────────────┐
│                        应用层                                │
│   retriever.py → search() → vector_store.search()           │
└─────────────────────────┬───────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                     向量数据库层                              │
│                    ChromaDB                                  │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │                    存储层                            │  │
│   │                                                     │  │
│   │  ┌──────────────┐    ┌──────────────────────────┐  │  │
│   │  │ SQLite       │    │ HNSW 索引 ← 就在这里     │  │  │
│   │  │ (文档/元数据) │    │ (向量检索加速)            │  │  │
│   │  └──────────────┘    └──────────────────────────┘  │  │
│   └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**简单说**：HNSW 是 ChromaDB 内部的一个**索引组件**，专门加速向量相似度搜索。

## 二、HNSW 的数据结构

### Q: HNSW 的具体数据结构是什么？

HNSW 是一个**多层图结构**，由以下文件组成：

```
data/chroma/3cc4021b-706b-457e-8296-799b75c12072/
├── data_level0.bin        # 向量数据（所有文档的 512 维浮点数）
├── link_lists.bin         # 图的连接关系（每个节点连接了哪些邻居）
├── header.bin             # 索引头信息（维度、节点数等）
├── length.bin             # 每个节点的邻居数量
└── index_metadata.pickle  # ID 映射表（文档 ID ↔ 内部标签）
```

**各文件详解**：

| 文件 | 内容 | 数据结构 | 作用 |
|------|------|----------|------|
| `data_level0.bin` | 所有向量 | `float32[663026][512]` | 存储原始向量数据 |
| `link_lists.bin` | 图连接 | `List[List[int]]` | 记录每个节点的邻居节点 |
| `header.bin` | 元信息 | 固定结构体 | 维度、最大层数、入口点等 |
| `length.bin` | 邻居数 | `int[663026]` | 每个节点在各层的邻居数量 |
| `index_metadata.pickle` | ID映射 | `dict` | ChromaDB ID ↔ HNSW 内部标签 |

### Q: 图结构长什么样？

```
HNSW 多层图结构示例（假设只有 8 个文档）：

Layer 2（最稀疏，只有少数节点）：
    A ──────────────────── D
    │                      │
    │                      │
Layer 1（中间层）：
    A ──── B ──── C ──── D ──── E
    │      │      │      │      │
    │      │      │      │      │
Layer 0（最密集，包含所有节点）：
    A ── B ── C ── D ── E ── F ── G ── H
    │    │    │    │    │    │    │    │
    └────┴────┴────┴────┴────┴────┴────┘
         （每个节点连接附近的邻居）

搜索过程：
1. 从最高层（Layer 2）的入口点开始
2. 在当前层找到离查询最近的节点
3. 跳到下一层，继续搜索
4. 到达 Layer 0 时，返回精确的 Top-K 近邻

复杂度：O(log n)，60 万条数据只需比较几百次
```

### Q: index_metadata.pickle 里存了什么？

```python
# 实际内容（我们刚才验证过的）
{
    "dimensionality": 512,                    # 向量维度
    "total_elements_added": 663026,           # 已添加的元素总数
    "max_seq_id": 663026,                     # 最大序列号
    "id_to_label": {                          # ChromaDB ID → HNSW 内部标签
        "民法典.txt_0": 0,
        "民法典.txt_1": 1,
        "刑法.txt_0": 2,
        ...
    },
    "label_to_id": {0: "民法典.txt_0", ...}, # 反向映射
    "id_to_seq_id": {...}                     # ID → 序列号映射
}
```

**作用**：当你查询时，ChromaDB 需要把文档 ID（如 `"民法典.txt_0"`）转换成 HNSW 内部的数字标签（如 `0`），才能在图中定位节点。

## 三、HNSW 用来干什么？

### Q: HNSW 的作用是什么？

**核心作用**：加速向量相似度搜索。

```
场景：在 663026 个 512 维向量中找到与查询最相似的 Top-5

方法 1：暴力搜索（Brute Force）
├── 逐一计算查询与每个向量的余弦相似度
├── 比较次数：663026 次
├── 时间复杂度：O(n × d) = O(663026 × 512)
└── 耗时：~100ms（太慢）

方法 2：HNSW 索引
├── 通过图结构快速跳转到相似区域
├── 比较次数：~500 次（只需访问图中的一小部分节点）
├── 时间复杂度：O(log n × d)
└── 耗时：~1ms（快 100 倍）
```

### Q: 什么时候需要 HNSW？

| 场景 | 是否需要 HNSW |
|------|---------------|
| 数据量 < 1000 | 不需要，暴力搜索足够快 |
| 数据量 1000 ~ 10000 | 可选，HNSW 稍快 |
| 数据量 > 10000 | 强烈推荐，差距明显 |
| **本项目（66 万条）** | **必须使用** |

## 四、为什么会出错？

### Q: HNSW 索引为什么会损坏？

**根本原因**：LeCaRD 数据入库过程中，进程被异常终止（可能是 OOM 或手动杀死）。

```
正常入库流程：
1. 计算文档向量
2. 添加到 ChromaDB
3. 更新 HNSW 索引（写入 data_level0.bin、link_lists.bin 等）
4. 保存 index_metadata.pickle（最后一步）

异常情况：
进程在步骤 3 或 4 中途被杀死
→ index_metadata.pickle 只写了一部分
→ 文件大小 10562001 字节，但 pickle 数据不完整
→ 下次读取时报错："trailing bytes found"
```

**错误信息解读**：
```
Error deserializing pickle file: eval error at offset 29609312: trailing bytes found
                                                         ↑
                                            文件在第 29609312 字节处截断
                                            后面还有数据没写完
```

### Q: 为什么删除 pickle 后查询仍失败？

**原因**：删除 `index_metadata.pickle` 后，ChromaDB 可能自动重建了一个不完整的映射表，导致：

1. **ID 映射丢失**：`id_to_label` 字典不完整
2. **查询时找不到节点**：当使用 `where` 过滤器时，ChromaDB 需要先通过 ID 找到节点，再检查元数据，这个过程失败了

```
查询流程（带 where 过滤）：
1. 用户查询 + where 条件
2. HNSW 搜索找到候选节点
3. 通过 ID 映射获取文档元数据
4. 检查元数据是否满足 where 条件 ← 这里失败
5. 返回过滤后的结果

失败原因：步骤 3 的 ID 映射不完整，导致 "Error finding id"
```

## 五、where 是什么？和向量有什么关系？

### Q: where 是操作数据库的操作吗？

**是的**，`where` 是 ChromaDB 的**元数据过滤条件**，类似于 SQL 的 `WHERE` 子句。

```python
# ChromaDB 的 where 查询
collection.query(
    query_embeddings=[query_vector],  # 向量查询
    n_results=5,                       # 返回 5 条
    where={"status": {"$ne": "repealed"}}  # ← where 过滤条件
)

# 等价的 SQL 概念
SELECT * FROM documents
WHERE status != 'repealed'           # ← where 过滤
ORDER BY cosine_similarity(embedding, query_vector)  # ← 向量排序
LIMIT 5
```

### Q: where 和向量有什么关系？

**where 和向量是两个独立的检索维度**，它们在查询中**同时生效**：

```
查询流程：

┌─────────────────────────────────────────────────────────┐
│                    collection.query()                    │
│                                                         │
│   输入：                                                │
│   ├── query_embeddings = [0.12, -0.34, ..., 0.56]      │
│   ├── n_results = 5                                     │
│   └── where = {"status": {"$ne": "repealed"}}           │
│                                                         │
│   处理：                                                │
│   1. HNSW 搜索：找到与 query_embedding 最相似的 Top-K   │
│      → 候选集：[doc_A, doc_B, doc_C, doc_D, ...]        │
│                                                         │
│   2. where 过滤：检查每个候选的元数据                    │
│      doc_A.status = "active"  → 保留 ✅                 │
│      doc_B.status = "repealed" → 过滤掉 ❌              │
│      doc_C.status = "active"  → 保留 ✅                 │
│                                                         │
│   3. 返回过滤后的 Top-5                                 │
└─────────────────────────────────────────────────────────┘
```

### Q: 为什么 where 过滤会失败？

**原因**：HNSW 索引损坏后，ID 映射不完整，导致：

1. **向量搜索本身可以工作**：HNSW 图结构还在，能找到相似向量
2. **但 where 过滤失败**：需要通过 ID 查找元数据时，映射表不完整

```
不带 where 的查询：
├── HNSW 搜索 → 返回向量相似度最高的 Top-K
├── 不需要查询元数据
└── 成功 ✅

带 where 的查询：
├── HNSW 搜索 → 返回候选集
├── 通过 ID 查找元数据 ← 这里失败
├── 过滤不满足条件的文档
└── 失败 ❌ "Error finding id"
```

## 六、修复方案

### Q: 如何修复 HNSW 索引？

**方案 A：重建集合（推荐）**

```python
# 1. 导出现有数据
collection = client.get_collection("legal_knowledge")
all_data = collection.get(include=["embeddings", "documents", "metadatas"])

# 2. 删除损坏的集合
client.delete_collection("legal_knowledge")

# 3. 创建新集合
new_collection = client.create_collection("legal_knowledge")

# 4. 重新导入（ChromaDB 会自动重建 HNSW 索引）
new_collection.add(
    ids=all_data["ids"],
    embeddings=all_data["embeddings"],
    documents=all_data["documents"],
    metadatas=all_data["metadatas"]
)
```

**方案 B：从源文件重新入库**

```bash
# 1. 删除损坏的数据
rm -rf data/chroma/

# 2. 重新入库
HF_ENDPOINT=https://hf-mirror.com PYTHONPATH=. uv run python scripts/ingest.py --split-mode legal data/datasets/scl/texts/
```

**方案 C：仅重建索引（最快）**

```python
# ChromaDB 可能会在下次启动时自动重建索引
# 或者触发手动 compaction
client.get_collection("legal_knowledge").modify()
```

## 七、相关概念对比

### Q: 向量检索 vs 关键词检索 vs 元数据过滤

| 维度 | 向量检索 | 关键词检索 | 元数据过滤 |
|------|----------|------------|------------|
| **原理** | 语义相似度 | 词频匹配 | 精确条件 |
| **实现** | HNSW 索引 | BM25 索引 | SQLite 查询 |
| **查询** | `query_embeddings` | `query_texts` | `where` |
| **擅长** | "夫妻义务" | "第1043条" | `status != 'repealed'` |
| **本项目** | ChromaDB | bm25s 库 | ChromaDB where |

### Q: 本项目的混合检索流程

```
用户查询："民法典第1043条"
        │
        ▼
┌───────────────────┐
│   retriever.py    │
│   search()        │
└───────┬───────────┘
        │
        ├────────────────────────────────────┐
        ▼                                    ▼
┌─────────────────┐                 ┌─────────────────┐
│   BM25 检索      │                 │   向量检索        │
│                 │                 │                 │
│ jieba 分词      │                 │ bge-small-zh    │
│ → ["民法典",    │                 │ → [0.12, ...]   │
│    "第1043条"]  │                 │                 │
│                 │                 │ ChromaDB        │
│ bm25s 检索      │                 │ .query()        │
│ → Top-10       │                 │ → Top-10        │
└────────┬────────┘                 └────────┬────────┘
         │                                   │
         └──────────────┬────────────────────┘
                        ▼
              ┌─────────────────┐
              │   RRF 融合排序   │
              │                 │
              │ score = Σ 1/(k+rank) │
              └────────┬────────┘
                       ▼
              ┌─────────────────┐
              │   最终 Top-5    │
              └─────────────────┘
```

**当前问题**：向量检索部分（`vector_store.search()`）在使用 `where` 过滤时失败，导致整个混合检索流程中断。

## 八、总结

| 问题 | 答案 |
|------|------|
| HNSW 是什么？ | 分层可导航小世界图，一种向量搜索算法 |

| where 和向量的关系？ | 两个独立维度：向量搜索 + 元数据过滤，同时生效 |
