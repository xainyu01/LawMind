## 一、Parquet 格式

### Q: Parquet 是什么？

Apache Parquet 是一种**列式存储**的二进制文件格式，专为大数据分析设计。

```
传统 CSV/JSON（行式存储）：          Parquet（列式存储）：
┌──────────────────────────┐     ┌──────────────────────────┐
│ id | name  | age | city  │     │ id:  [1, 2, 3, 4, ...]   │
│ 1  | 张三  | 25  | 北京  │     │ name: [张三, 李四, ...]   │
│ 2  | 李四  | 30  | 上海  │     │ age:  [25, 30, 28, ...]   │
│ 3  | 王五  | 28  | 广州  │     │ city: [北京, 上海, ...]    │
└──────────────────────────┘     └──────────────────────────┘
读取 name 列 → 扫描所有行           读取 name 列 → 直接定位列块
```

### Q: 什么时候使用 Parquet？

| 场景 | 使用 Parquet | 使用 JSON |
|------|-------------|-----------|
| HuggingFace 数据集分发 | ✅ 默认格式 | 可选 |
| 大规模数据分析（Spark/Pandas） | ✅ 首选 | 不适合 |
| LLM 训练数据 | 有时用 | ✅ 更常见 |
| API 数据交换 | ❌ 不适合 | ✅ 首选 |
| 配置文件 | ❌ | ✅ 首选 |
| 流式读取 | ❌ | ✅ 逐行读取 |

### Q: 为什么 HuggingFace 用 Parquet？

1. **压缩率高**：列式存储 + 压缩算法，文件体积比 JSON 小 50-80%
2. **读取快**：只读需要的列，不用解析整行
3. **类型安全**：schema 内嵌，不会出现 "age" 字段有时是数字有时是字符串
4. **生态兼容**：Pandas、Spark、DuckDB 等直接读取

```python
# Parquet 读取示例
import pandas as pd
df = pd.read_parquet("data.parquet")          # 一行代码
# 只读某一列（极快）
titles = pd.read_parquet("data.parquet", columns=["title"])
```

### Q: Parquet vs JSON 对比

| 特性 | Parquet | JSON |
|------|---------|------|
| 存储方式 | 列式、二进制 | 行式、文本 |
| 文件大小 | 小（压缩率高） | 大（无压缩） |
| 读取速度 | 快（列裁剪） | 慢（全量解析） |
| 类型信息 | 有 schema | 无 schema |
| 可读性 | 二进制，不可直接阅读 | 文本，可直接阅读 |
| 流式处理 | 不支持 | ✅ 逐行读取 |
| 适用规模 | GB~TB 级 | KB~MB 级 |

## 三、本项目的分片（Chunking）逻辑

### 两种分段模式

#### 1. 默认模式（`--split-mode default`）

使用 LangChain `RecursiveCharacterTextSplitter`，按递归分隔符切割：

```
chunk_size = 500 字符
chunk_overlap = 50 字符（前后片段重叠，避免语义断裂）

分隔符优先级：
  \n\n（段落） → \n（换行） → 。（句号） → ；（分号） → ，（逗号） → 空格
```

**原理**：先尝试按段落切，如果一段超过 500 字，再按句号切，依此类推。

```
原文（1200字）→ 切成 3 段：
  [段1: 0-500字] [段2: 450-950字] [段3: 900-1200字]
                 ↑ 与段1重叠50字    ↑ 与段2重叠50字
```

#### 2. 法律模式（`--split-mode legal`）

按法条结构精确分段，保持法条完整性：

```python
# 法律法规：按"第X条"分段
正则: r'(第[一二三四五六七八九十百千零\d]+条)'

# 判决书：按关键段落分段
正则: r'(?=本院认为|判决如下|如不服本判决|依照)'

# 合同：按"第X条"分段（同法律法规）
```

**示例**：

```
原文：
  第一千零四十二条 禁止包办、买卖婚姻和其他干涉婚姻自由的行为...
  第一千零四十三条 夫妻应当互相忠实，互相尊重，互相关爱...

分段结果：
  片段1: "第一千零四十二条 禁止包办、买卖婚姻..." (article=第一千零四十二条)
  片段2: "第一千零四十三条 夫妻应当互相忠实..."   (article=第一千零四十三条)
```

### 分段后每个片段的元数据

```python
{
    "source": "民法典节选.txt",      # 来源文件
    "chunk_index": 42,               # 片段序号
    "article": "第一千零四十三条",    # 法条编号（法律模式）
    "doc_type": "law",               # 文档类型
    "effective_date": "2021-01-01",  # 生效日期
    "status": "active"               # 时效状态（active/repealed）
}
```

### 为什么 chunk_size=500？

| chunk_size | 优点 | 缺点 |
|-----------|------|------|
| 200 | 精确匹配 | 语义不完整，丢失上下文 |
| **500** | **平衡精确度和完整性** | - |
| 1000 | 上下文丰富 | 检索时噪音多，embedding 质量下降 |
| 2000+ | 完整文档 | 向量检索基本失效 |

法律条文通常 100-300 字/条，500 字可以完整包含一条法条 + 部分上下文。

## 五、文档读取实现

### Q: 项目中 Word、PDF、TXT 分别怎么读取？

三种格式使用不同的 LangChain Document Loader：

```python
# document_loader.py
from langchain_community.document_loaders import (
    PyPDFLoader,      # PDF
    Docx2txtLoader,   # Word
    TextLoader,       # TXT
)

def load_document(file_path: str) -> List[Document]:
    ext = Path(file_path).suffix.lower()
    if ext == ".pdf":
        loader = PyPDFLoader(file_path)        # 使用 PyPDF2 库
    elif ext in (".docx", ".doc"):
        loader = Docx2txtLoader(file_path)     # 使用 docx2txt 库
    elif ext == ".txt":
        loader = TextLoader(file_path, encoding="utf-8")  # 直接读取
    else:
        raise ValueError(f"不支持的文件格式: {ext}")
    return loader.load()
```

### Q: 每种格式的读取原理是什么？

| 格式 | 库 | 原理 | 返回内容 |
|------|-----|------|----------|
| **PDF** | PyPDF2 | 解析 PDF 内部结构，提取文本流 | 每页一个 Document |
| **Word** | docx2txt | 解析 .docx（本质是 ZIP 包含 XML），提取段落文本 | 整个文件一个 Document |
| **TXT** | 内置 open() | 直接读取文本文件 | 整个文件一个 Document |

**PDF 读取细节**：
```
PDF 文件结构：
┌─────────────────────────┐
│ Header                   │  ← 版本信息
│ Body                     │  ← 页面、字体、文本流、图片等对象
│ Cross-Reference Table    │  ← 对象索引
│ Trailer                  │  ← 文件尾
└─────────────────────────┘

PyPDF2 读取流程：
1. 解析 PDF 结构
2. 遍历每一页
3. 提取页面中的文本流（TextObject）
4. 拼接为字符串
5. 包装为 Document(page_content="...", metadata={"page": N})
```

**Word 读取细节**：
```
.docx 文件结构（本质是 ZIP）：
├── [Content_Types].xml
├── word/
│   ├── document.xml      ← 主要内容（段落、表格）
│   ├── styles.xml        ← 样式
│   └── media/            ← 图片
└── _rels/

docx2txt 读取流程：
1. 解压 ZIP
2. 解析 word/document.xml
3. 提取所有 <w:p> 段落的文本
4. 拼接为字符串
5. 包装为 Document
```

### Q: 如果 PDF 是图片（扫描版）怎么办？

**当前不支持**。`PyPDFLoader` 只能提取文本型 PDF 中的文字。如果 PDF 是扫描版（图片），将返回空内容。

**识别方法**：
- 文本型 PDF：可以选择、复制文字
- 扫描型 PDF：文字是图片的一部分，无法选择

**解决方案**：需要集成 OCR（光学字符识别）

```python
# 方案 1：使用 Unstructured（推荐）
from langchain_community.document_loaders import UnstructuredFileLoader

loader = UnstructuredFileLoader(file_path, strategy="ocr_only")

# 方案 2：使用 PaddleOCR（中文精度高）
from paddleocr import PaddleOCR
ocr = PaddleOCR(lang="ch")
result = ocr.ocr("scanned_page.png")
```

## 六、ChromaDB 向量数据库

### Q: ChromaDB 是什么样的数据库？

ChromaDB 是一个**向量数据库**，专门存储和检索高维向量。与传统关系数据库不同：

```
传统数据库（MySQL）：              向量数据库（ChromaDB）：
┌────────────────────────┐      ┌────────────────────────┐
│ id │ name │ age │ city │      │ id │ embedding │ text  │
│ 1  │ 张三 │ 25  │ 北京 │      │ 1  │ [0.1,0.3, │ "夫妻…│
│ 2  │ 李四 │ 30  │ 上海 │      │    │  ...,0.8] │       │
└────────────────────────┘      │ 2  │ [0.2,0.5, │ "禁止…│
                                │    │  ...,0.7] │       │
查询：WHERE age > 25             └────────────────────────┘
返回：精确匹配的行                查询：给定一个向量，找最相似的
                                 返回：按相似度排序的文档
```

### Q: ChromaDB 的存储结构是什么？

```
ChromaDB 存储结构：
┌─────────────────────────────────────────────────────────┐
│                    ChromaDB                              │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│  │ Collection   │  │ Collection   │  │ Collection   │    │
│  │ "legal_know" │  │ "other"      │  │ ...          │    │
│  └──────┬──────┘  └─────────────┘  └─────────────┘    │
│         │                                               │
│  ┌──────▼──────────────────────────────────────────┐   │
│  │              内部存储                             │   │
│  │                                                  │   │
│  │  ┌──────────────┐  ┌──────────────────────────┐ │   │
│  │  │ SQLite 数据库 │  │ HNSW 索引文件            │ │   │
│  │  │              │  │                          │ │   │
│  │  │ - embeddings │  │ - data_level0.bin (向量) │ │   │
│  │  │ - documents  │  │ - link_lists.bin (图连接) │ │   │
│  │  │ - metadatas  │  │ - header.bin             │ │   │
│  │  │ - segments   │  │ - index_metadata.pickle  │ │   │
│  │  └──────────────┘  └──────────────────────────┘ │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**SQLite 存储**：文档 ID、文本内容、元数据、嵌入队列
**HNSW 索引**：向量数据、图结构（用于快速相似度搜索）

### Q: HNSW 索引是什么？为什么需要它？

HNSW（Hierarchical Navigable Small World）是一种**近似最近邻（ANN）搜索算法**。

**问题**：如果有 60 万个 512 维向量，要找到与查询向量最相似的 Top-K 个：
- **暴力搜索**：逐一比较 60 万次，O(n) 复杂度，太慢
- **HNSW 索引**：构建一个多层图结构，O(log n) 复杂度，极快

```
HNSW 索引结构（多层图）：

Layer 2（稀疏层）：  A ──────────── D
                     │              │
Layer 1（中间层）：  A ── B ── C ── D ── E
                     │    │    │    │    │
Layer 0（密集层）：  A ── B ── C ── D ── E ── F ── G ── H

搜索过程：
1. 从最顶层（Layer 2）开始，找到最近的节点
2. 跳到下一层（Layer 1），在更密集的图中继续搜索
3. 到达最底层（Layer 0），找到精确的近邻

优点：搜索复杂度 O(log n)，60 万条数据只需比较几百次
```

**HNSW 索引包含的文件**：

| 文件 | 内容 | 作用 |
|------|------|------|
| `data_level0.bin` | 所有向量数据 | 存储原始 512 维浮点数 |
| `link_lists.bin` | 图的连接关系 | 记录每个节点连接了哪些邻居 |
| `header.bin` | 索引头信息 | 维度、节点数等元信息 |
| `index_metadata.pickle` | ID 映射表 | 文档 ID ↔ 内部标签的映射 |

### Q: pickle 文件为什么会损坏？

**原因**：入库过程中进程被终止（OOM 或手动杀死），pickle 文件写入不完整。

```
正常写入：
index_metadata.pickle = pickle.dump({
    "dimensionality": 512,
    "total_elements_added": 600000,
    "id_to_label": {"doc1": 1, "doc2": 2, ...},  ← 大量数据
    "max_seq_id": ...
})

异常情况：
进程被杀死 → pickle.dump() 中途停止 → 文件只写了一部分
→ 下次读取时：pickle.load() 报错 "trailing bytes found"
```

**修复方法**：
1. 删除损坏的 pickle 文件（临时修复）
2. 重建整个 HNSW 索引（根本修复）

### Q: 向量检索的完整流程是什么？

```
用户查询："夫妻之间有什么义务"
        │
        ▼
┌───────────────────┐
│ 1. 文本 → 向量     │  bge-small-zh 模型
│    "夫妻之间..."   │  → [0.12, -0.34, ..., 0.56]  (512维)
└───────┬───────────┘
        │
        ▼
┌───────────────────┐
│ 2. HNSW 索引搜索   │  在 60 万个向量中找最相似的 Top-K
│    比较余弦相似度   │  cosine_similarity(query, doc)
└───────┬───────────┘
        │
        ▼
┌───────────────────┐
│ 3. 返回结果        │  按相似度排序
│    + 文档文本      │  "第一千零四十三条 夫妻应当..."
│    + 元数据        │  {source: "民法典.txt", article: "第1043条"}
│    + 相似度分数    │  distance: 0.3031
└───────────────────┘
```

### Q: 384 维 vs 512 维是怎么回事？

**背景**：
- ChromaDB 自带默认嵌入函数 `all-MiniLM-L6-v2`，产生 **384 维**向量
- 本项目使用 `bge-small-zh`，产生 **512 维**向量

**问题场景**：
```python
# 错误方式：使用 ChromaDB 默认函数（384 维）
col.query(query_texts=["测试"])  # ← 内部用 384 维函数

# 正确方式：使用我们的嵌入函数（512 维）
embedding = bge_model.encode("测试")  # ← 512 维
col.query(query_embeddings=[embedding])  # ← 传入已计算的向量
```

**Collection 创建时的维度**：
```python
# 首次创建 collection 时，ChromaDB 记录维度
collection = client.get_or_create_collection("legal_knowledge")
# 此时维度未确定，直到第一次 add() 时才固定

# 第一次 add() 使用 512 维向量 → collection 维度固定为 512
collection.add(ids=["1"], embeddings=[[0.1]*512])

# 之后 query() 必须也是 512 维，否则报错
collection.query(query_embeddings=[[0.2]*384])  # ❌ 维度不匹配
```

**之前的维度**：本项目从始至终使用 `bge-small-zh`（512 维），不存在"之前是 384 维"的情况。384 维是 ChromaDB 默认函数的维度，仅在直接使用 `query_texts` 时触发。

**解决方案**：始终通过 `query_embeddings` 参数传入我们自己计算的向量，不依赖 ChromaDB 默认函数。本项目的代码已经是这样做的。

## 七、BM25 关键词检索

### Q: BM25 是什么？

BM25（Best Matching 25）是一种经典的**关键词检索算法**，基于 TF-IDF 改进。

**核心思想**：一个词在文档中出现得越多（TF 高），且在整个语料库中出现得越少（IDF 高），则该词对这篇文档越重要。

```
BM25 公式简化：
score(q, d) = Σ IDF(qi) * (f(qi, d) * (k1 + 1)) / (f(qi, d) + k1 * (1 - b + b * |d|/avgdl))

其中：
- IDF(qi): 词 qi 的逆文档频率（越稀有的词权重越高）
- f(qi, d): 词 qi 在文档 d 中的出现频率
- k1, b: 调节参数（通常 k1=1.5, b=0.75）
- |d|: 文档长度
- avgdl: 平均文档长度
```

### Q: BM25 在本项目中的作用是什么？

**作用**：与向量检索互补，提供精确关键词匹配能力。

```
场景对比：

查询："民法典第1043条"
├── BM25 检索：直接匹配"第1043条"这个关键词 → 精准命中 ✅
└── 向量检索：语义理解"夫妻义务"相关 → 可能命中，但不精确

查询："夫妻之间有什么义务"
├── BM25 检索：匹配"夫妻""义务"关键词 → 可能漏掉"互相忠实"等表述
└── 向量检索：语义理解 → 精准命中"第一千零四十三条" ✅
```

**总结**：
- BM25 擅长：精确匹配（法条编号、专有名词、数字）
- 向量检索擅长：语义理解（自然语言描述、同义词、近义表达）

### Q: BM25 和向量检索是什么关系？

**互补关系**，两者各有优势，通过 RRF 融合取长补短：

```
                    ┌──────────────────┐
                    │   用户查询        │
                    │ "夫妻之间有什么义务"│
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼                              ▼
    ┌─────────────────┐          ┌─────────────────┐
    │   BM25 检索      │          │   向量检索        │
    │                  │          │                  │
    │ 1. jieba 分词    │          │ 1. bge-small-zh  │
    │    "夫妻/之间/   │          │    编码为 512 维  │
```

### Q: BM25 在本项目中是怎么运行的？

**运行流程**：

```python
# 1. 构建 BM25 索引（首次启动或数据更新时）
# retriever.py:125
all_data = vector_store.get_all_documents()  # 从 ChromaDB 获取所有文档
corpus = all_data["documents"]                # 文档文本列表

# 2. jieba 中文分词
tokens = [list(jieba.cut(text)) for text in corpus]
# 例："夫妻应当互相忠实" → ["夫妻", "应当", "互相", "忠实"]

# 3. 构建 BM25 索引
bm25 = bm25s.BM25()
bm25.index(tokens)  # 计算每个词的 TF-IDF 权重

# 4. 持久化到磁盘（避免每次重启重建）
bm25.save("data/chroma/bm25_cache/bm25_idx.pkl")

# 5. 检索时
query_tokens = list(jieba.cut("夫妻之间有什么义务"))
# → ["夫妻", "之间", "有", "什么", "义务"]

results, scores = bm25.retrieve(query_tokens, k=5)
# → 返回最相关的 5 个文档的索引和分数
```

**索引持久化**：
```
data/chroma/bm25_cache/
├── bm25_idx.pkl      # BM25 索引（词 → 文档的 TF-IDF 权重）
├── doc_ids.pkl       # 文档 ID 列表（索引位置 → ChromaDB ID）
└── doc_count.txt     # 文档总数（用于判断是否需要重建）
```

### Q: RRF 融合算法是什么？

RRF（Reciprocal Rank Fusion）是一种简单有效的排序融合算法：

```python
# RRF 公式
score(doc) = Σ 1 / (k + rank_i)

# 其中：
# - k = 60（常数，避免排名靠前的分数过高）
# - rank_i = 文档在第 i 个检索器中的排名（1-indexed）
```

**示例**：
```
文档 A：BM25 排名第 1，向量排名第 3
  RRF = 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323

文档 B：BM25 排名第 3，向量排名第 1
  RRF = 1/(60+3) + 1/(60+1) = 0.0159 + 0.0164 = 0.0323

文档 C：BM25 排名第 1，向量未命中
  RRF = 1/(60+1) = 0.0164

→ 文档 A 和 B 排名相同（都在两个检索器中有较好排名）
→ 文档 C 排名较低（只在一个检索器中命中）
```

## 八、嵌入模型与向量

### Q: 什么是嵌入（Embedding）？

嵌入是将文本转换为**固定长度的数值向量**的过程。向量的每一维捕捉文本的某种语义特征。

```
文本："夫妻应当互相忠实"
        │
        ▼ bge-small-zh 模型
向量：[0.12, -0.34, 0.56, ..., 0.78]  (512 个浮点数)

文本："夫妻之间有什么义务"
        │
        ▼ bge-small-zh 模型
向量：[0.11, -0.32, 0.54, ..., 0.76]  (512 个浮点数)

两个向量很接近 → 语义相似
```

### Q: 为什么用 512 维？维度越高越好吗？

| 维度 | 模型 | 精度 | 存储 | 检索速度 |
|------|------|------|------|----------|
| 384 | all-MiniLM-L6-v2 | 一般 | 小 | 快 |
| **512** | **bge-small-zh** | **良好** | **中** | **快** |
| 768 | bge-base-zh | 更好 | 大 | 中 |
| 1024 | bge-large-zh | 最好 | 更大 | 慢 |

**权衡**：维度越高 → 语义表达越丰富 → 但存储和计算成本越高

**本项目选择 512 维的原因**：
1. 中文法律文本检索精度足够
2. 模型只有 ~100MB，适合 8GB 显存
3. 检索速度快（HNSW 索引在 512 维下表现良好）

### Q: 余弦相似度是什么？

向量检索的核心是计算**余弦相似度**（Cosine Similarity）：

```
cosine_similarity(A, B) = (A · B) / (|A| * |B|)

取值范围：[-1, 1]
  1  → 完全相同方向（语义完全一致）
  0  → 正交（无关）
 -1  → 完全相反（语义相反）
```

**示例**：
```
查询向量：[0.12, -0.34, 0.56]
文档A：  [0.11, -0.32, 0.54]  → 相似度 0.99（非常相似）
文档B：  [0.80, 0.20, -0.10]  → 相似度 0.15（不太相关）

→ 返回文档 A（相似度更高）
```
